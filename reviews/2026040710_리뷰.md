## 2. 잘한 점

### 주문 관리 기능을 모듈로 잘 분리했어요!

`admin/pages/components/order/` 폴더를 보면 각 파일이 하나의 역할만 담당하고 있어요. 이것이 바로 **높은 응집도**의 좋은 예시예요!

| 파일 | 역할 |
|------|------|
| `orderListApi.js` | API 호출만 |
| `renderRows.js` | 테이블 그리기만 |
| `paginationButton.js` | 페이지 버튼만 |
| `calcListNum.js` | 건수 계산만 |
| `search.js` | 검색 필터만 |
| `showDetails.js` | 상세 이동만 |
| `index.js` | 이것들을 조합 |

이렇게 파일을 나누면 "검색 기능을 고치고 싶을 때 search.js만 열면 된다"는 장점이 있어요. 이전 리뷰에서 설명한 "응집도는 높고 결합도는 낮게" 원칙을 잘 실천하고 있어요!

### `paginationButton.js`의 인터페이스가 깔끔해요

```javascript
// paginationButton.js
export function paginationButton(container, totalPages, currentPage, onPageChange) {
```

이 함수는 **콜백 패턴**을 사용해서 결합도를 낮췄어요. `paginationButton`은 "페이지가 바뀌면 `onPageChange`를 호출해줘"라고만 하고, 실제로 무슨 일이 벌어지는지는 신경 쓰지 않아요. 이것이 **낮은 결합도**의 좋은 예시예요!

`index.js`에서 이렇게 연결하고 있죠:
```javascript
paginationButton(buttonComponents, totalPages, currentPage, renderOrderList);
```

### `renderRows.js`에서 DOM API를 적극 활용했어요

`createElement` + `appendChild` 패턴으로 테이블을 만든 것은 좋은 습관이에요. `innerHTML`로 문자열을 만드는 것보다 안전하고, 요소 하나하나를 코드로 제어할 수 있어요.

### API 호출에서 토큰 인증을 구현했어요

```javascript
// orderListApi.js
const token = localStorage.getItem("token");
if (!token) {
  console.error("토큰이 없습니다.");
  return null;
}
```

토큰이 없을 때 에러 처리를 해주는 것이 좋아요. `Bearer` 토큰을 Authorization 헤더에 담는 것도 표준적인 방식이에요.

### 대시보드 디자인이 깔끔하고 반응형까지 적용했어요

`admin-dashboard.html`에서 데스크톱에서는 사이드바, 모바일에서는 드롭다운 메뉴로 전환되는 반응형 레이아웃이 잘 구현되어 있어요. `xl:flex`, `xl:hidden` 같은 반응형 클래스를 적절하게 사용했어요.

### 베스트셀러 섹션이 추가되어 메인 페이지가 완성도 있어요

`index.html`에 베스트셀러 섹션이 추가되면서 상품 쇼핑몰다운 구성이 갖춰졌어요. 모바일에서는 작은 카드, 데스크톱에서는 큰 카드로 보이는 반응형 처리도 잘했어요.

## 3. 개선하면 좋을 점

### 3-1. `search.js`에서 이벤트 리스너가 매 페이지마다 중복 등록돼요 — 메모리 누수 (중요도: 높음)

**현재 상태:**

```javascript
// admin/pages/components/order/index.js 45번째 줄
// renderOrderList()가 호출될 때마다 search()도 호출돼요
search(orders, orderSearch, (filteredOrders) => {
  renderRows(tbody, filteredOrders);
});
```

```javascript
// admin/pages/components/order/search.js 4번째 줄
export function search(orders, input, onSearch) {
  if (!input) return;
  input.addEventListener("input", () => {   // ← 매번 새 리스너 추가!
    // ...
  });
}
```

페이지 1을 보고 → 페이지 2를 클릭하면 `renderOrderList(2)`가 호출되고, 그 안에서 `search()`가 다시 호출돼요. `search()` 안에서 `addEventListener`가 또 실행되니까, **검색창에 이벤트 리스너가 계속 쌓여요**.

5페이지를 이동하면 리스너가 5개, 10페이지를 이동하면 10개... 이것을 **메모리 누수**라고 해요.

**왜 개선하면 좋을까? (응집도 관점)**

`search()` 함수가 "이벤트 등록"과 "필터링"이라는 두 가지 역할을 동시에 하고 있어요. "이벤트 등록"은 처음 한 번만, "필터링"은 매번 실행해야 하는데, 이 두 가지가 섞여 있어서 문제가 생기는 거예요.

> **쉽게 말하면:** 택배함에 "새 택배가 오면 알려줘"라고 알림을 설정하는 것은 한 번이면 충분해요. 그런데 매일 아침마다 같은 알림을 또 설정하면, 택배 한 개가 왔을 때 알림이 100번 울리겠죠?

**추천:**

```javascript
// search.js — "이벤트 등록"은 한 번만, "필터링 대상 데이터"는 갱신 가능하게
let currentOrders = [];

export function initSearch(input, onSearch) {
  if (!input) return;
  input.addEventListener("input", () => {
    const keyword = input.value.trim().toLowerCase();
    if (!keyword) {
      onSearch(currentOrders);
      return;
    }
    const filtered = currentOrders.filter((item) => {
      const name = item.shippingAddress?.recipientName?.toLowerCase() ?? "";
      const number = item.orderNumber?.toLowerCase() ?? "";
      return name.includes(keyword) || number.includes(keyword);
    });
    onSearch(filtered);
  });
}

export function updateSearchData(orders) {
  currentOrders = orders;
}
```

```javascript
// index.js에서
import { initSearch, updateSearchData } from "./search.js";

// 처음 한 번만 이벤트 등록
initSearch(orderSearch, (filteredOrders) => {
  renderRows(tbody, filteredOrders);
});

async function renderOrderList(page = 1) {
  // ... API 호출 후
  updateSearchData(orders);   // 데이터만 갱신
  renderRows(tbody, orders);
}
```

### 3-2. `search.js`에서 null 값에 `.toLowerCase()`를 호출하면 에러가 나요 (중요도: 높음)

**현재 상태:**

```javascript
// admin/pages/components/order/search.js 13~14번째 줄
const recipientName = item.shippingAddress.recipientName.toLowerCase() ?? "";
const orderNumber = item.orderNumber.toLowerCase() ?? "";
```

**왜 문제가 될까?**

JavaScript에서 `??` 연산자(Nullish Coalescing)는 **왼쪽 값이 `null`이나 `undefined`일 때** 오른쪽 값을 사용해요. 그런데 여기서는 `.toLowerCase()`가 `??`보다 **먼저** 실행돼요!

```
실행 순서:
1. item.shippingAddress.recipientName   → null 일 수 있어요!
2. null.toLowerCase()                   → 💥 에러 발생!
3. ?? ""                                → 여기까지 도달 못해요
```

만약 API에서 받은 주문 데이터에 `shippingAddress`가 없거나 `recipientName`이 `null`이면, **페이지가 멈춰요**.

> **쉽게 말하면:** "안전벨트를 매고 → 출발"해야 하는데, "출발하고 → 안전벨트 매기"를 하고 있어요. 순서가 바뀌면 사고 났을 때 안전벨트가 소용없어요.

**추천:**

```javascript
// 수정 전 (위험!)
const recipientName = item.shippingAddress.recipientName.toLowerCase() ?? "";

// 수정 후 (안전!)
const recipientName = (item.shippingAddress?.recipientName ?? "").toLowerCase();
const orderNumber = (item.orderNumber ?? "").toLowerCase();
```

`??`로 먼저 빈 문자열을 보장한 다음에 `.toLowerCase()`를 호출하면 안전해요.

### 3-3. `admin-login.html`에 비밀번호가 그대로 노출되어 있어요 (중요도: 높음)

**현재 상태:**

```html
<!-- admin/pages/admin-login.html 60번째 줄 -->
<p class="mt-6 text-center text-xs text-gray-400">
  테스트 계정: admin&#64;gentlemonster.com / admin123
</p>
```

```javascript
// admin/pages/admin-login.html 77번째 줄 (인라인 스크립트)
if (email === "admin@gentlemonster.com" && password === "admin123") {
  window.location.href = "admin-dashboard.html";
}
```

두 가지 문제가 있어요:

1. **비밀번호가 HTML 소스에 그대로** 보여요. 브라우저에서 "소스 보기"를 하면 누구나 볼 수 있어요.
2. **로그인 성공해도 토큰을 저장하지 않아요**. 대시보드에서는 주문 API를 호출할 때 `localStorage.getItem("token")`으로 토큰을 읽는데, 로그인 시 토큰을 저장하는 코드가 없어요.

**왜 개선하면 좋을까? (결합도 관점)**

지금 로그인 페이지와 대시보드/주문 페이지의 인증 방식이 **서로 연결되지 않아요**:
- 로그인 페이지: 하드코딩 비교 → `admin-dashboard.html`로 이동만
- 주문 페이지: `localStorage`에서 토큰을 읽어서 API 호출

이 둘이 같은 "인증 흐름"인데 서로 다른 방식으로 동작하고 있어서 **기능의 응집도가 낮아요**.

> **쉽게 말하면:** 놀이공원에서 입장권을 사는데, 정문에서는 "얼굴만 보고" 들여보내주고, 놀이기구에서는 "입장권을 보여주세요"라고 해요. 입장할 때 입장권을 안 줬으니 놀이기구를 탈 수 없겠죠?

**추천:**

```javascript
// admin/pages/admin-login.html 스크립트 부분
if (email === "admin@gentlemonster.com" && password === "admin123") {
  // 로그인 성공 시 토큰 저장 (나중에 API 인증 서버가 생기면 서버에서 받아요)
  localStorage.setItem("token", "test-admin-token");
  btn.innerHTML = "로그인 성공";
  setTimeout(() => {
    window.location.href = "admin-dashboard.html";
  }, 500);
}
```

그리고 대시보드에서 토큰이 없으면 로그인 페이지로 되돌려보내는 코드를 추가하면 더 좋아요:

```javascript
// admin-dashboard.html 스크립트 맨 위
if (!localStorage.getItem("token")) {
  window.location.href = "admin-login.html";
}
```

### 3-4. `admin-login.js`가 아무 곳에서도 사용되지 않아요 — 죽은 코드 (중요도: 높음)

**현재 상태:**

```javascript
// admin/pages/admin-login.js
import "../style.css";

const form = document.getElementById("admin-login-form");

form.addEventListener("submit", async (e) => {
  e.preventDefault();
  const email = document.getElementById("email").value;
  const password = document.getElementById("password").value;
  console.log(email, password);
});
```

이 파일은 **어디에서도 `import`되거나 `<script>`로 로드되지 않아요**. `admin-login.html`은 인라인 `<script>`에 `handleLogin()` 함수를 직접 작성해서 사용하고 있어요.

또한 `admin-login-form`이라는 ID의 폼을 찾고 있지만, `admin-login.html`의 폼 ID는 `loginForm`이에요. 같은 기능을 두 번 만든 셈이에요.

**왜 개선하면 좋을까? (응집도 관점)**

### 3-5. `admin-dashboard.html`이 CDN Tailwind를 사용하고 있어요 — 빌드 시스템 불일치 (중요도: 높음)

**현재 상태:**

```html

<script>
  tailwind.config = {
    theme: {
      extend: {
        colors: {
          "admin-gray": "#667085",
          "admin-border": "#e4e7ec",
          // ...
        },
      },
    },
  };
</script>
```

프로젝트의 다른 페이지들은 모두 `vite.config.js` + `@tailwindcss/vite` 플러그인을 사용하고 있어요:

```javascript
// vite.config.js
import tailwindcss from '@tailwindcss/vite'
export default defineConfig({
  plugins: [tailwindcss()],
})
```

**왜 개선하면 좋을까? (결합도 관점)**

같은 프로젝트에서 **두 가지 다른 방식**으로 Tailwind를 사용하면:

1. **색상이 달라질 수 있어요**: `src/components/style.css`에 정의한 `--admin-gray`와 대시보드의 `admin-gray`가 서로 다른 값이에요 (`#6A7282` vs `#667085`)
2. **빌드할 때 대시보드만 빠져요**: `npm run build`를 실행하면 Vite가 처리하는 파일만 빌드되는데, CDN을 쓰는 대시보드는 빌드 파이프라인에 포함되지 않아요
3. **CDN Tailwind는 개발용이에요**: 공식 문서에서도 "프로덕션에서는 사용하지 마세요"라고 안내하고 있어요

> **쉽게 말하면:** 반 친구들은 모두 같은 교복을 입고 다니는데, 한 명만 비슷한 색의 사복을 입고 와서 "교복이에요"라고 하는 거예요. 멀리서 보면 비슷하지만 자세히 보면 달라요.

**추천:**

CDN 스크립트와 인라인 config를 제거하고, 프로젝트의 Tailwind 설정을 사용하세요:

```html
<!-- admin/pages/admin-dashboard.html -->
<head>
  <!-- 이것을 삭제 -->
  <!-- <script src="https://cdn.tailwindcss.com"></script> -->
  <!-- <script>tailwind.config = { ... }</script> -->

  <!-- 이것을 추가 -->
  <link rel="stylesheet" href="../../src/style.css" />
  <link rel="stylesheet" href="../../src/components/style.css" />
</head>
```

커스텀 색상은 `src/components/style.css`의 CSS 변수를 활용하면 돼요.

### 3-8. `calcListNum.js`가 API 응답 구조에 과도하게 의존하고 있어요 — 결합도 높음 (중요도: 중간)

**현재 상태:**

```javascript
// admin/pages/components/order/calcListNum.js
export function calcListNum(data, range, currentPage) {
    const pageSize = 20;
    const totalCount = data.data.pagination?.totalCount ?? 0;
    // ...
}
```

```javascript
// admin/pages/components/order/index.js 43번째 줄
calcListNum(data, range, currentPage);
```

`index.js`에서 이미 `data.data.pagination`을 분석해서 `totalCount`, `totalPages`, `currentPage`를 꺼내놓았는데, `calcListNum`에는 전체 `data` 객체를 그대로 넘기고 있어요. 그래서 `calcListNum`이 `data.data.pagination?.totalCount`라는 **API 응답의 깊은 구조**를 알아야 해요.

**왜 개선하면 좋을까? (결합도 관점)**

`calcListNum.js`의 역할은 "몇 건 중 몇 번째를 보여주는지 계산"하는 거예요. 이 기능에 필요한 것은 `totalCount`와 `currentPage`뿐인데, API 응답의 전체 구조(`data.data.pagination`)를 알아야 한다는 것은 **불필요한 결합**이에요.

나중에 API 응답 형식이 바뀌면 (`data.result.pagination`으로 바뀐다면?) `index.js`와 `calcListNum.js` 둘 다 수정해야 해요.

> **쉽게 말하면:** 친구에게 "3층 302호 김철수에게 편지 전해줘"라고 하면, 친구가 건물 구조를 알아야 해요. "김철수에게 편지 전해줘"라고만 하면 친구는 주소록만 보면 돼요.

**추천:**

```javascript
// 수정 전 (data 전체를 받음)
export function calcListNum(data, range, currentPage) {
    const totalCount = data.data.pagination?.totalCount ?? 0;
}

// 수정 후 (필요한 값만 받음)
export function calcListNum(totalCount, range, currentPage) {
    const pageSize = 20;
    const start = (currentPage - 1) * pageSize + 1;
    const end = Math.min(currentPage * pageSize, totalCount);
    range.textContent = `${start} - ${end}`;
}
```

```javascript
// index.js에서 호출할 때
calcListNum(totalCount, range, currentPage);   // 이미 꺼내둔 값을 전달
```

### 3-12. `admin-login.html`에서 인라인 이벤트 핸들러를 사용하고 있어요 (중요도: 낮음)

**현재 상태:**

```html
<!-- admin/pages/admin-login.html 21번째 줄 -->
<form id="loginForm" onsubmit="handleLogin(event)" novalidate>
```

**왜 개선하면 좋을까?**

`onsubmit="handleLogin(event)"` 같은 인라인 이벤트 핸들러는 **HTML 초기에 사용하던 오래된 방식**이에요. 모던 JavaScript에서는 `addEventListener`를 사용하는 것이 권장돼요. 이유:

1. HTML(구조)과 JavaScript(동작)가 섞여서 **응집도가 낮아져요**
2. 보안 정책(CSP: Content Security Policy)에서 인라인 핸들러를 차단할 수 있어요
3. 이벤트를 추가하거나 제거하기 어려워요

```javascript
// 인라인 대신 addEventListener 사용
const form = document.getElementById("loginForm");
form.addEventListener("submit", handleLogin);
```

## 6. 마무리

이번에 관리자 페이지가 본격적으로 만들어지기 시작했어요! 주문 관리 기능에서 **파일을 역할별로 분리한 것**이 정말 훌륭해요. `orderListApi.js → index.js → renderRows.js` 흐름을 보면 "API 호출 → 데이터 가공 → 화면 렌더링"이라는 역할이 명확하게 나뉘어 있어요. 이것이 바로 응집도 높은 코드예요!

`paginationButton.js`에서 콜백 패턴을 사용한 것도 인상적이에요. 이런 패턴을 `search.js`에도 적용하면 이벤트 리스너 중복 문제를 해결할 수 있어요.

지금 가장 시급한 것은 **search.js의 null 에러**와 **이벤트 리스너 누수**를 고치는 거예요. 두 가지 모두 실제로 사용자가 페이지를 쓸 때 문제가 될 수 있어요. 해결 가이드를 보면서 하나씩 따라해보세요!

그리고 CDN Tailwind와 Vite Tailwind 혼용 문제도 중요해요. 같은 프로젝트에서 같은 방식으로 스타일링을 하면 나중에 배포할 때 고생하지 않아요.
