## 2. 잘한 점

### API 계층과 UI 계층을 분리한 것이 훌륭해요!

관리자 페이지에서 API 호출 함수와 화면 렌더링 로직을 **별도 파일로 분리**한 것이 정말 좋은 시도예요. 예를 들어:

```
API/order/orderListApi.js     → 서버에서 데이터를 가져옴
components/order/renderRows.js → 가져온 데이터로 화면을 그림
components/order/index.js      → 두 파일을 연결하는 조립 역할
```

이렇게 하면 **API URL이 바뀌어도** renderRows.js를 수정할 필요가 없고, **화면 디자인이 바뀌어도** API 파일을 수정할 필요가 없어요. 이것이 바로 **"낮은 결합도"**예요!

### 사용자 모듈에서 주문 모듈의 공용 함수를 재사용했어요

`user/index.js`에서 `paginationButton`과 `calcListNum`을 `../order/` 폴더에서 가져와 쓰고 있어요:

```javascript
// admin/pages/components/user/index.js
import { paginationButton } from "../order/paginationButton.js";
import { calcListNum } from "../order/calcListNum.js";
```

같은 기능을 복사하지 않고 **import로 재사용**한 것은 정말 좋은 판단이에요! 코드 중복을 줄이는 가장 기본적이고 효과적인 방법이에요.

### Bearer 토큰 인증을 올바르게 구현했어요

모든 API 함수에서 `localStorage`에서 토큰을 가져와 `Authorization: Bearer ${token}` 헤더에 넣는 인증 흐름이 정확해요:

```javascript
const token = localStorage.getItem("token");
if (!token) { console.error("토큰이 없습니다."); return null; }
const res = await fetch(url, {
  headers: { Authorization: `Bearer ${token}` }
});
```

"토큰이 없으면 API를 호출하지 않고 일찍 리턴"하는 **Guard 패턴**도 잘 적용했어요.

### try-catch로 에러 처리를 빠짐없이 했어요

모든 API 함수와 `index.js` 엔트리 파일에서 `try-catch`로 에러를 감싸고 있어요. 네트워크 요청은 언제든 실패할 수 있기 때문에, 이 습관이 정말 중요해요.

### 관리자 로그인 페이지의 `<label>`과 `<input>` 연결이 정확해요

```html
<!-- admin-login.html -->
<label class="block text-sm text-gray-700 mb-1" for="email">이메일</label>
<input id="email" type="email" ... />
```

`for="email"`과 `id="email"`이 정확히 연결되어 있어요. 이전 리뷰에서 지적한 "label이 없는 input" 문제를 관리자 로그인에서는 잘 해결했어요!

### 관리자 목록 페이지에 `<header>`와 `<main>` 시맨틱 태그를 사용했어요

```html
<!-- orders.html, users.html -->
<header class="flex flex-col lg:flex-row ...">
  <h1>주문 관리</h1>
  ...
</header>
<main class="p-8">
  ...
</main>
```

이전 리뷰에서 "시맨틱 태그가 없다"고 했던 것을 관리자 페이지에서는 잘 적용했어요. 이 좋은 패턴을 메인 페이지에도 적용해보세요!

### null 체크 습관이 생기고 있어요

`showDetails` 함수에서 `if (!button) return;`과 `if (userId == null)` 체크를 넣고 있고, `user/index.js`에서 `if (!data || !data.data)` 체크를 하고 있어요. 이전 리뷰에서 배운 **null 체크 패턴**이 자리잡아가고 있어서 좋아요!

## 3. 개선하면 좋을 점

### 3-1. 사용자 상세보기가 주문 페이지로 이동하는 버그예요 (중요도: 높음)

**현재 상태:**

```javascript
// admin/pages/components/user/showDetails.js 9번째 줄
button.addEventListener("click", () => {
  window.location.href = `/admin/orders/${userId}`;  // ← 주문 페이지?!
});
```

**왜 개선하면 좋을까?**

사용자 관리 페이지에서 눈 모양(👁) 버튼을 클릭하면 **사용자 상세 페이지**로 이동해야 하는데, 지금은 **주문 페이지 URL**인 `/admin/orders/${userId}`로 이동하고 있어요. 이건 `order/showDetails.js`를 복사하면서 URL을 수정하지 않은 것으로 보여요.

이 버그 때문에 관리자가 사용자 정보를 확인하려고 클릭하면 엉뚱한 페이지로 가거나, 404 에러가 나요.

**추천:**

```javascript
// 수정 전
window.location.href = `/admin/orders/${userId}`;

// 수정 후 — 사용자 상세 페이지로 이동
window.location.href = `/admin/pages/user-details.html?userId=${userId}`;
```

> 아직 `user-details.html`이 없다면, 먼저 그 페이지를 만들거나, 임시로 `alert("사용자 ID: " + userId)`를 띄워서 연결을 확인해보세요.

### 3-2. 사용자 이름 합치기에 연산자 우선순위 버그가 있어요 (중요도: 높음)

**현재 상태:**

```javascript
// admin/pages/components/user/renderRows.js 17번째 줄
tdId.textContent = item.lastName + item.firstName ?? "-";
```

**왜 개선하면 좋을까?**

JavaScript에서 `+` 연산자는 `??` 연산자보다 **먼저 실행**돼요. 그래서 위 코드는 이렇게 동작해요:

```javascript
// 실제 실행 순서
tdId.textContent = (item.lastName + item.firstName) ?? "-";
//                  ↑ 이 부분이 먼저 계산됨

// item.lastName이 undefined이면?
// undefined + "홍길동" = "undefined홍길동" 😱
// "undefined홍길동" ?? "-" = "undefined홍길동" (null이 아니니까 ?? 무시!)
```

`??` 연산자는 왼쪽이 `null` 또는 `undefined`일 때만 오른쪽을 사용해요. 그런데 `undefined + "홍길동"`은 문자열 `"undefined홍길동"`이 되어버리기 때문에 `??`는 동작하지 않아요!

> **쉽게 말하면:** 수학에서 곱셈이 덧셈보다 먼저인 것처럼, `+`가 `??`보다 먼저예요. 괄호로 순서를 명확히 해야 해요.

**추천:**

```javascript
// 수정 전
tdId.textContent = item.lastName + item.firstName ?? "-";

// 수정 후 — 각각 null 체크 후 합치기
tdId.textContent = (item.lastName ?? "") + (item.firstName ?? "") || "-";
```

이렇게 하면:
- `lastName`이 없으면 `""` (빈 문자열)
- `firstName`이 없으면 `""` (빈 문자열)
- 둘 다 없으면 `""` → `||` 때문에 `"-"` 표시

### 3-3. 주문 검색에서 null 값이 오면 에러가 나요 (중요도: 높음)

**현재 상태:**

```javascript
// admin/pages/components/order/search.js 13~14번째 줄
const recipientName = item.shippingAddress.recipientName.toLowerCase() ?? "";
const orderNumber = item.orderNumber.toLowerCase() ?? "";
```

**왜 개선하면 좋을까?**

이 코드에는 두 가지 문제가 있어요:

**문제 1:** `recipientName`이 `null`이면 `.toLowerCase()`를 호출하는 순간 에러가 나요. `??`는 `.toLowerCase()` **이후**에 실행되기 때문에 이미 늦어요.

```javascript
// null.toLowerCase()  → ❌ TypeError!
// TypeError가 발생한 후에는 ?? 까지 도달하지 못해요
```

**문제 2:** `.toLowerCase()`가 성공하면 항상 문자열을 반환해요. 문자열은 `null`도 `undefined`도 아니니까 `??`는 절대 동작하지 않아요.

재미있는 것은, **같은 기능**인 `user/search.js`에서는 올바르게 작성되어 있어요:

```javascript
// user/search.js — 올바른 방법! 👍
const userId = String(item.userId ?? "").toLowerCase();
```

**추천:**

```javascript
// 수정 전 (order/search.js)
const recipientName = item.shippingAddress.recipientName.toLowerCase() ?? "";
const orderNumber = item.orderNumber.toLowerCase() ?? "";

// 수정 후 — user/search.js와 동일한 패턴 적용
const recipientName = String(item.shippingAddress?.recipientName ?? "").toLowerCase();
const orderNumber = String(item.orderNumber ?? "").toLowerCase();
```

`?.`(옵셔널 체이닝)을 `shippingAddress` 뒤에도 넣어야 `shippingAddress` 자체가 없을 때도 안전해요.

### 3-6. API 함수 5개가 거의 똑같은 코드를 반복하고 있어요 — 응집도를 높이면 좋아요 (중요도: 중간)

**현재 상태:**

5개의 API 파일(`orderListApi.js`, `orderDetailsApi.js`, `orderPatchApi.js`, `userListApi.js`, `userDeleteApi.js`)에서 이 패턴이 **동일하게 반복**되고 있어요:

```javascript
// 5개 파일 모두 이 구조가 똑같음!
export async function someApi(...) {
  try {
    const token = localStorage.getItem("token");       // ← 5번 반복
    if (!token) {                                       // ← 5번 반복
      console.error("토큰이 없습니다.");                  // ← 5번 반복
      return null;                                      // ← 5번 반복
    }                                                   // ← 5번 반복
    const res = await fetch(url, {                      // ← 5번 반복
      headers: { Authorization: `Bearer ${token}` }     // ← 5번 반복
    });                                                 // ← 5번 반복
    if (!res.ok) { throw new Error("API 요청 실패"); }   // ← 5번 반복
    const data = await res.json();                      // ← 5번 반복
    return data;                                        // ← 5번 반복
  } catch (error) {                                     // ← 5번 반복
    console.error(error);                               // ← 5번 반복
    return null;                                        // ← 5번 반복
  }                                                     // ← 5번 반복
}
```

**왜 개선하면 좋을까? (응집도 & 결합도 관점)**

지금은 "API 인증 로직"이 5개 파일에 **흩어져(결합도 높음)** 있어요. 만약 토큰 만료 처리를 추가하거나, `Authorization` 헤더 형식이 바뀌면 5개 파일을 모두 수정해야 해요.

"인증 처리"라는 하나의 관심사를 한 곳에 모으면(**응집도 높게**), 나머지 API 함수들은 "어떤 URL에 어떤 요청을 보낼까"만 신경 쓰면 돼요.

> **쉽게 말하면:** 카페에서 주문할 때마다 매번 지갑에서 카드를 꺼내고, 결제하고, 카드를 넣는 과정을 반복하는 대신, **카카오페이 같은 결제 앱** 하나를 만들어두면 편하겠죠? `fetchWithAuth`가 바로 그 결제 앱이에요.

**추천:**

```javascript
// admin/pages/API/fetchWithAuth.js — 공용 인증 함수 (새 파일)
const BASE_URL = "https://api.fullstackfamily.com/api/gentlelion/v1/admin";

export async function fetchWithAuth(endpoint, options = {}) {
  const token = localStorage.getItem("token");
  if (!token) {
    console.error("토큰이 없습니다.");
    return null;
  }

  try {
    const res = await fetch(`${BASE_URL}${endpoint}`, {
      ...options,
      headers: {
        Authorization: `Bearer ${token}`,
        ...options.headers,
      },
    });

    if (!res.ok) throw new Error(`API 요청 실패: ${res.status}`);
    return await res.json();
  } catch (error) {
    console.error(error);
    return null;
  }
}
```

이제 각 API 파일이 아주 짧아져요:

```javascript
// admin/pages/API/order/orderListApi.js — 수정 후
import { fetchWithAuth } from "../fetchWithAuth.js";

export function orderAPI(page) {
  return fetchWithAuth(`/orders?page=${page}`);
}
```

```javascript
// admin/pages/API/order/orderPatchApi.js — 수정 후

export function orderPatchApi(id, data) {
  return fetchWithAuth(`/orders/${id}/status`, {
    method: "PATCH",
    headers: { "Content-Type": "application/json" },

  });

```

5개 파일이 각각 3~5줄로 줄어들고, 인증 로직은 **한 곳**에서만 관리하면 돼요!

### 3-8. 검색 이벤트 리스너가 페이지를 넘길 때마다 계속 쌓여요 (중요도: 중간)

**현재 상태:**

```javascript
// admin/pages/components/order/index.js (요약)
async function renderOrderList(page = 1) {
  const data = await orderAPI(page);
  // ... 행 렌더링, 페이지네이션 ...

  search(data.data.orders, orderSearch, (filtered) => {   // ← 매번 새 리스너 추가!
    renderRows(tbody, filtered);
  });
}
renderOrderList(1);  // 1페이지 로드 → search 리스너 1개
// 2페이지 클릭 → renderOrderList(2) → search 리스너 2개!
// 3페이지 클릭 → renderOrderList(3) → search 리스너 3개!!
```

**왜 개선하면 좋을까?**

`search()` 함수는 `input.addEventListener("input", ...)`을 호출해요. 페이지를 넘길 때마다 `renderOrderList()`이 다시 실행되면서 **새 이벤트 리스너가 추가**되지만, **이전 리스너는 제거되지 않아요**.

10페이지를 넘기면 검색 입력창에 글자를 칠 때마다 **10개의 함수가 동시에 실행**돼요. 성능이 나빠지고, 이전 페이지의 데이터로 검색한 결과와 현재 페이지 결과가 섞일 수 있어요.

> **쉽게 말하면:** 알람 앱에서 "매일 7시" 알람을 10번 설정하면, 7시에 알람이 10번 울리겠죠? 새 알람을 설정하기 전에 이전 알람을 지워야 해요.

**추천:**

```javascript
// order/search.js — 수정 후
let currentHandler = null;   // 이전 리스너를 기억하는 변수

export function search(orders, input, onSearch) {
  if (!input) return;

  // 이전 리스너가 있으면 제거
  if (currentHandler) {
    input.removeEventListener("input", currentHandler);
  }

  // 새 리스너를 만들어서 등록
  currentHandler = () => {
    const keyword = input.value.trim().toLowerCase();
    if (!keyword) { onSearch(orders); return; }
    const filtered = orders.filter((item) => {
      const recipientName = String(item.shippingAddress?.recipientName ?? "").toLowerCase();
      const orderNumber = String(item.orderNumber ?? "").toLowerCase();
      return recipientName.includes(keyword) || orderNumber.includes(keyword);
    });
    onSearch(filtered);
  };

  input.addEventListener("input", currentHandler);
}
```

`user/search.js`에도 같은 수정을 적용하세요.

이 파일을 삭제하거나, `admin-login.html`의 인라인 스크립트를 이 파일로 옮기세요. 사용하지 않는 파일이 프로젝트에 남아 있으면 다른 팀원이 혼란스러워할 수 있어요.

### 3-12. 삭제 버튼에 모두 같은 `id="delete"`가 지정되어 있어요 (중요도: 낮음)

**현재 상태:**

```javascript
// admin/pages/components/user/renderRows.js 50번째 줄
const deletebutton = document.createElement("button");
deletebutton.id = "delete";   // ← 모든 행에서 같은 ID!
```

**왜 개선하면 좋을까?**

HTML에서 `id`는 **페이지 전체에서 딱 하나만 존재**해야 해요. 사용자가 10명이면 `id="delete"`인 버튼이 10개가 되는데, 이건 HTML 표준을 위반하는 거예요.

지금은 문제가 안 보이지만, 나중에 `document.getElementById("delete")`로 특정 버튼을 찾으려고 하면 **첫 번째 버튼만** 찾아져요.

또한 `deletebutton`이라는 변수명과 `detailButton`이라는 변수명의 **대소문자 규칙(camelCase)**이 다른 것도 살짝 아쉬워요.

**추천:**

```javascript
// 수정 전
const deletebutton = document.createElement("button");
deletebutton.id = "delete";

// 수정 후 — id 대신 class 사용, camelCase 통일
const deleteButton = document.createElement("button");   // ← camelCase
deleteButton.className = "delete-btn cursor-pointer";     // ← id 대신 class
```

관리자 페이지를 **처음부터 직접 설계하고 구현**한 것이 정말 대단해요! 특히 다음 부분들이 인상적이에요:

지금 가장 중요한 과제는 **3가지 버그 수정**(URL 잘못, 이름 합치기, 검색 null)이에요. 이 세 가지는 각각 1~2줄만 수정하면 되니까 금방 고칠 수 있어요!
